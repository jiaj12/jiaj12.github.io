kubernetes Helm 包管理工具 java 客户端 microbean-helm 使用指南

microBean Helm project 工具使用 java 与服务端的 Helm 组件通信

在 java 项目中，我们本来不得不使用 helm 命令行工具来进行 helm 相关操作。现在，通过这个工具，java 应用就可以使用 Helm 的特性如： charts 和 releases 来管理 kubernetes 应用。

版本

microBean Helm 项目的版本和 helm 还有 tiller 的版本一一对应。例如，版本号为 2.8.2.1.1.0 的 microBean Helm 版本号为 2.8.2 ，非生成的(SemVer-compatible) 版本为 1.1.0

文档

microBean Helm project 的文档连接。

Helm

Helm 是 Kubernetes 的应用包管理工具。它包括了一个命令行工具 helm ，一个名叫 Tiller 的服务端组件。 helm 就是一个 Tiller 客户端。

Tiller

Tiller 是 Helm 的服务端组件。Tiller使用Helm charts 包Kubernetes manifest templates模板和他们的配置。 together with their values. microBean Helm 可以帮助您 通过 java 构建和使用这些 charts 还有他们的 releases 然后，将这些 charts 等配置文件与 Tiller 交互。

安装Tiller

一般来说，TillerIn a normal Helm usage scenario, Tiller is 被 helm 命令行工具 (通过 helm init 子命令)实时安装。在 Kubernetes 集群中作为一个 Pod 运行。 microBean Helm 提供了 TillerInstaller class ，可以通过 Java 来实现同样的功能。

Tiller 连接

因为 Tiller 通常作为 Pod 运行，我们不太容易在集群外部直接访问。 helm 命令行工具在集群内将本地端口转发到 Tiller Pod 中的端口。通过隧道和 Tiller 服务端建立连接。microBean Helm 项目通过 Java 类库实现了相同功能。

Tiller 通信

Tiller 本质上是一个 gRPC 应用。microBean Helm 生成 Java 依赖 对于 gRPC API, 允许 程序通过 Java 类和 Tiller 通信。

ReleaseManager

一般来说，安装和升级 Release 的操作逻辑被封装在 Tiller 服务端内部。但是这种方法在 Java 项目中行不通。 helm 命令行工具在安装时处理Helm Chart的 requirements.yaml 文件，决定将怎样的命令发送给 Tiller。因此，如果您使用 microBean Helm project，您必须使用非自动生成的 ReleaseManager 类 来执行 Helm相关操作，因为我们将一部分 Helm 功能放在了这里。

Chart.Builder 和 ChartOrBuilder

Google’s Protocol Buffers—which underlies the Tiller API, and hence substantial portions of the microbean-helm project—makes a distinction between builders and messages. Chart.Builder 是Chart的构建器， Chart.Builder 和Chart 是 ChartOrBuilder 接口的实现类。这些类是通过protocol buffers 的源代码直接生成的，所以它们的名字和形态都不可更改。

通常，如果只关心 Tiller-related 对象的 “读” 操作，您应该关注 &hellip;OrBuilder 接口。如果是在构建一些东西，应该使用 &hellip;.Builder 相关类。
